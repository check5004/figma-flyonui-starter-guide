✅ Figmaコンポーネント設計の6つの基本原則

1. 整理された命名規則と階層構造

スラッシュ(/)でレイヤー名を区切り、アセットパネルでコンポーネントを自動グループ化します。

Button/Primary/Large/Default

これにより、Assetsパネルで階層的に表示されます。

2. バリアントとプロパティの活用

類似コンポーネントはバリアントとして統合し、ブーリアンプロパティやインスタンススワッププロパティで柔軟性を高めます。

  • State: Default, Hover, Active
  • Size: Small, Medium, Large
  • Icon: Boolean (表示/非表示)

3. Auto Layoutの適用

柔軟なレイアウトを実現し、コンテンツの変化に自動で対応させます。

  • Padding、Gap、Resizingを適切に設定
  • Min/Max Widthでレスポンシブ対応

4. アトミックデザインの考え方

最小単位(Atoms)から段階的にコンポーネントを構築します。

Atoms → Molecules → Organisms → Templates → Pages

5. SSOT(Single Source of Truth)

スタイルはローカルスタイル(Color Styles、Text Styles)として定義し、一貫性を保ちます。

変数(Variables)も活用して、テーマ切り替えに対応。

6. チームライブラリの活用

コンポーネントをチーム全体で共有し、作業効率と一貫性を向上させます。

Publishして他のファイルから参照できるようにします。

🔄 FlyonUIハイブリッド運用戦略(超重要!)

🤔 初心者が陥りがちな誤解

❌ 間違った認識:

  • 「FlyonUIのコンポーネントをそのまま使えばOK」
  • 「コピーしてデタッチすれば自由にカスタムできる」
  • 「とりあえず全部FlyonUIから始めて、後で考えよう」

⚠️ これらは後で大きな問題を引き起こします!

✨ 正解:ハイブリッド戦略

🎯 戦略の核心

FlyonUIのコード基盤との整合性を保ちつつ、管理の容易さとカスタマイズの柔軟性を両立させる

📦 2ファイル構成

1 01_DesignSystem - Master

FlyonUIライブラリへの依存をここに集約。FlyonUIのインスタンスをデタッチせず、そのインスタンスをラップ(コンポーネント化)して、自社独自のコンポーネントライブラリを作成。

2 02_ProductUI

実際の画面デザインはこちらで行う。参照するライブラリは自社の01_DesignSystemファイルのみ。これにより、FlyonUIライブラリへの直接アクセスを遮断し、依存関係をシンプルに。

🔧 ハイブリッド戦略の実装手順

  • 初期セットアップ
    • 01_DesignSystem - Masterファイルを作成
    • FlyonUIライブラリを有効化
    • 基本スタイル(カラー、タイポグラフィ)を定義
  • ProductUIでの作業開始
    • 02_ProductUIファイルを作成
    • まずはFlyonUIライブラリを有効化
    • FlyonUIのインスタンスを直接使用して画面設計開始
  • カスタムが必要になった時
    • 01_DesignSystemで該当コンポーネントをラップ
    • FlyonUIのインスタンスを配置
    • それをコンポーネント化(デタッチしない!)
    • 必要ならバリアントプロパティを追加
  • ProductUIでの置き換え
    • 02_ProductUIで自社ライブラリを有効化
    • FlyonUIインスタンスを自社コンポーネントに置換
    • 段階的に移行していく
  • 最終段階
    • 02_ProductUIでFlyonUIライブラリを無効化
    • 自社ライブラリのみを参照
    • クリーンな依存関係の完成!

💡 重要なポイント

  • 段階的なラップ:最初から全てのコンポーネントをラップする必要はありません。カスタムが必要になったものから順次対応。
  • デタッチ禁止:カスタムが必要な場合も、デタッチではなくバリアントプロパティとして追加。
  • 一貫性の維持:デフォルト版もカスタム版も01_DesignSystemで一元管理。
  • アップデート対応:FlyonUIのアップデートも自動的に反映される。

🤖 FigmaMCPでAIレビューを活用する

🔌 FigmaMCPとは?

FigmaMCPは、Figmaデスクトップアプリと連携して、選択したコンポーネントの情報をAI(Claude)が直接読み取れるようにするツールです。これにより、命名規則やコンポーネント構造を自動的にチェックできます。

セットアップガイド: FigmaMCP 利用ガイド(公式)

※ Figmaデスクトップアプリのインストールが必要です

📋 使い方の流れ

1. Figmaで選択

レビューしたいコンポーネントやフレームをFigmaデスクトップアプリで選択します。

2. プロンプトをコピー

以下のプロンプトをコピーして、ClaudeDesktopやCursorに貼り付けます。

3. AIが自動分析

ClaudeがFigmaMCPを通じてコンポーネント情報を読み取り、自動的に分析します。

4. 改善提案を受け取る

命名規則やベストプラクティスに基づいた具体的な改善提案が得られます。

🎯 プロンプト集

💎 包括的コンポーネントレビュー

コンポーネント全体を徹底的にチェックする包括的なプロンプトです。初めてレビューする際や、重要なコンポーネントの品質を確認したい時に使用します。

現在Figmaで選択しているコンポーネントを、以下の観点で包括的にレビューしてください:

【チェック項目】

1. **命名規則の確認**
   - スラッシュ(/)区切りで階層構造になっているか
   - Category/Type/Size/Stateのパターンに従っているか
   - 英語で統一されているか
   - 略語を使用していないか

2. **コンポーネント構造**
   - Auto Layoutが適切に設定されているか
   - Padding、Gap、Resizingの設定は適切か
   - 不要なネストがないか
   - レイヤー名は分かりやすいか

3. **バリアント設計**
   - プロパティ名は適切か(State, Size, Typeなど)
   - バリアントの組み合わせに漏れや重複はないか
   - Boolean型プロパティを活用できているか

4. **スタイルの一貫性**
   - Color StylesやText Stylesを使用しているか
   - ハードコーディングされた値はないか
   - Variablesを活用しているか

5. **FlyonUIとの整合性**
   - FlyonUIのコンポーネントを適切にラップしているか
   - デタッチせずにインスタンスのまま使用しているか

【出力形式】
- ✅ 良い点
- ⚠️ 改善が必要な点
- 💡 具体的な改善提案(ステップバイステップで)
- 📝 推奨される命名例

最後に、このコンポーネントの品質スコア(0-100)と総合評価をお願いします。

🏷️ 命名規則チェック(クイック版)

命名規則だけを素早くチェックしたい時に使用します。大量のコンポーネントを効率的にレビューする際に便利です。

選択中のコンポーネント/レイヤーの命名をチェックしてください:

【チェックポイント】
✓ スラッシュ(/)区切りの階層構造
✓ Category/Type/Size/Stateのパターン
✓ 英語表記の統一
✓ 略語の使用禁止
✓ 見た目ではなく機能で命名

【現在の命名】
→ (Claudeが自動で読み取ります)

【改善案】
もし命名に問題があれば、以下の形式で提案してください:
- ❌ 現在: [現在の名前]
- ✅ 推奨: [改善後の名前]
- 理由: [なぜその命名が良いのか]

問題なければ「✅ 命名規則に準拠しています」と返答してください。

📐 Auto Layout構造チェック

Auto Layoutの設定を詳しくチェックし、レスポンシブ性と保守性を評価します。

選択中のコンポーネントのAuto Layout設定をレビューしてください:

【確認項目】

1. **Auto Layoutの適用状況**
   - 適用されているか
   - Direction(横/縦)は適切か
   - ネストしたAuto Layoutが適切に設定されているか

2. **スペーシング**
   - Paddingの値は適切か
   - Gapの設定は統一されているか
   - マジックナンバーではなく変数を使用しているか

3. **Resizing設定**
   - Hug/Fixed/Fillの選択は適切か
   - Min/Max Widthが設定されているか
   - レスポンシブに対応できる構造か

4. **子要素の配置**
   - Alignment(揃え)は適切か
   - 不要なAbsolute positioningはないか

【出力】
- 現在の設定の要約
- ⚠️ 問題点
- 💡 最適化の提案
- 🎯 推奨設定値

コンテンツが変化しても破綻しない構造になっているかを重点的に評価してください。

🎨 バリアント構造チェック

バリアントを持つコンポーネントの設計品質を評価します。

選択中のコンポーネントのバリアント構造を分析してください:

【分析項目】

1. **プロパティ設計**
   - プロパティ名は適切か(State, Size, Type, Icon など)
   - Boolean型を活用できているか
   - Instance Swapプロパティは適切に使われているか

2. **バリアントの完全性**
   - 必要なバリアントの組み合わせは全て揃っているか
   - 不要な重複バリアントはないか
   - 状態遷移(Default → Hover → Active など)は論理的か

3. **メンテナンス性**
   - 新しいバリアントを追加しやすい構造か
   - プロパティ数は適切か(多すぎないか)
   - 命名規則は統一されているか

4. **実装連携**
   - 開発者が使いやすい構造か
   - propsとして渡しやすいプロパティ名か

【出力形式】
📊 現在のバリアント構成
- プロパティ一覧
- バリアント数

⚠️ 問題点
- 不足しているバリアント
- 不要なバリアント
- プロパティ設計の課題

💡 改善提案
- 追加すべきバリアント
- 統合すべきバリアント
- プロパティの再設計案

最後に「拡張性スコア」(0-100)を評価してください。

🔄 FlyonUI統合チェック

FlyonUIのコンポーネントを適切に活用できているかをチェックします。ハイブリッド戦略に沿った運用ができているかを確認します。

選択中のコンポーネントについて、FlyonUIとの統合状況をチェックしてください:

【確認ポイント】

1. **ラップ戦略の実装**
   - FlyonUIのインスタンスをデタッチせずに使用しているか
   - 適切にコンポーネント化(ラップ)されているか
   - カスタマイズはバリアントプロパティで実現しているか

2. **依存関係の管理**
   - このコンポーネントはどのファイルに配置すべきか
     * 01_DesignSystem - Master
     * 02_ProductUI
   - ライブラリの参照は適切か

3. **カスタマイズ方法**
   - デタッチが必要な箇所はあるか(ある場合は要注意)
   - バリアントで対応できない部分はあるか
   - FlyonUIの更新に追従できる構造か

4. **代替案の検討**
   - FlyonUIのコンポーネントを使うべきか
   - ゼロから作るべきか
   - ラップするべきか

【出力】
✅ 適切な点
⚠️ リスクがある点
💡 推奨される対応
🎯 最適なファイル配置

ハイブリッド戦略の観点から、長期的なメンテナンス性を重視して評価してください。

💡 効果的な使い方のコツ

  • 定期的なレビュー: 新しいコンポーネントを作成したら、すぐにレビューする習慣をつけましょう。
  • スラッシュコマンド化: ClaudeDesktopやCursorのスラッシュコマンドに登録すると、さらに効率的です。
  • チームで共有: これらのプロンプトをチーム全体で共有し、品質基準を統一しましょう。
  • 段階的改善: 全ての指摘を一度に修正しようとせず、優先度をつけて対応しましょう。
  • 学習ツールとして: AIの指摘を読むことで、ベストプラクティスを学ぶことができます。

🔧 トラブルシューティング

Q: Claudeがコンポーネント情報を読み取れない
A: FigmaMCPが正しくセットアップされているか、Figmaデスクトップアプリでコンポーネントが選択されているかを確認してください。

Q: レビュー結果が表示されない
A: プロンプト全体がコピーされているか、ClaudeのコンテキストウィンドウがFigmaに接続されているかを確認してください。

Q: 改善提案が具体的でない
A: プロンプトに「具体的な例を示して」「ステップバイステップで」などの指示を追加してみてください。

🏷️ 命名規則:迷わないための黄金ルール

基本パターン

要素 命名パターン 例
コンポーネント Category/Type/Size/State Button/Primary/Large/Default
バリアント Property=Value State=Hover, Size=Medium
変数 category.type.name color.primary.500

FlyonUI対応の命名例

01_DesignSystem - Master/ ├── 📄 Foundations │ ├── Color Styles (Primary, Secondary, etc.) │ ├── Text Styles (Heading/H1, Body/Regular, etc.) │ └── Variables (color.primary.500, spacing.md, etc.) │ ├── 📄 Atoms │ ├── Button/Primary/Large/Default │ ├── Button/Primary/Large/Hover │ ├── Input/Text/Medium/Default │ └── Badge/Status/Success │ ├── 📄 Molecules │ ├── Form-Field/Text/Default │ ├── Card/Product/Horizontal │ └── Nav-Item/Default/Active │ └── 📄 Organisms ├── Header/Dashboard/Desktop ├── Sidebar/Navigation/Expanded └── Table/Data/Sortable 02_ProductUI/ ├── 📄 01_Dashboard ├── 📄 02_User-Management ├── 📄 03_Reports └── 📄 04_Settings

⚠️ よくある命名の失敗

  • ❌ btn-primary-lg → 略語は避ける
  • ❌ Button_Primary_Large → アンダースコアではなくスラッシュ
  • ❌ ボタン/プライマリ/大 → 英語で統一
  • ❌ Button/blue/big → 色や見た目ではなく機能で命名

✅ 正しい命名:

  • ✅ Button/Primary/Large/Default
  • ✅ Input/Text/Medium/Disabled
  • ✅ Card/Product/Horizontal/Hover

👁️ このセクションについて

📍 使い方:

  • ホバー: 各要素にマウスを重ねると、関連するコンポーネントが黄色でハイライトされます
  • クリック: 要素をクリックすると、濃い黄色で固定ハイライトされ、詳細を確認できます
  • 階層理解: 下から上へ（Atoms → Files）の構築順序と、各レベルの役割・依存関係を理解できます

🎯 Atomic Designとは

Atomic Designは、UIを化学の原子・分子の概念に基づいて階層的に構築する設計手法です。最小単位（Atoms）から段階的に組み合わせることで、一貫性があり保守しやすいデザインシステムを構築できます。

📚 階層構造（下から上へ構築）

  1. ⚛️ Atoms（原子） - 最小単位の要素（ボタン、アイコン、テキストなど）
  2. 🔹 Molecules（分子） - Atomsの組み合わせ（検索フォーム、メニュー項目など）
  3. 🔷 Organisms（有機体） - MoleculesとAtomsで構成される複雑な部品（ヘッダー、サイドバーなど）
  4. 📋 Templates - Organismsを配置したレイアウト（ワイヤーフレーム）
  5. 📄 Pages - 実際のコンテンツを含む完成画面
  6. 📁 Files - Figmaでの管理単位（DesignSystem, ProductUI）

  • 階層構造: Atomic Designに基づいた、Atoms → Molecules → Organisms → Templates → Pagesの階層がどのように組み立てられるかを視覚的に理解できます。

🎯 Figmaでの実装時のポイント

  • Atomsから構築: まずは最小単位(Logo、Icon、Labelなど)をコンポーネント化します。
  • Moleculesで組み合わせ: AtomsをまとめてMolecules(Nav Item、Menu Itemなど)を作成します。
  • Organismsで機能単位: Moleculesを組み合わせて機能ブロック(Header、Sidebar、Tableなど)を構築します。
  • バリアントで管理: 同じコンポーネントの異なる状態はバリアントプロパティで切り替えます。
  • 命名規則の徹底: 各レベルで一貫した命名規則を使用することで、チーム全体での理解が深まります。

📝 ステップバイステップ実装ガイド

🚀 実装の順序とベストプラクティス

Atomic Designは下から上へ（Atoms → Files）構築します。各レベルで以下のポイントに注意してください。

Step 1: ⚛️ Atoms の作成

作業場所: 01_DesignSystem - Master ファイル

目的: 最小単位の再利用可能なコンポーネントを定義

📋 実装手順
  1. FlyonUIのコンポーネントから使用するものを選定
  2. FlyonUIインスタンスを配置し、コンポーネント化（デタッチしない）
  3. 必要なバリアントプロパティを定義：
    • State: Default, Hover, Active, Disabled
    • Size: Small, Medium, Large
    • Type: Primary, Secondary, Tertiary
  4. 命名規則を適用: {カテゴリ}/{タイプ}/{サイズ}/{状態}
  5. Color StylesとText Stylesを使用（ハードコード禁止）

💡 ポイント: FlyonUIにないAtomsが必要な場合は、自作します。その場合もバリアントとスタイルの一貫性を保ちます。

Step 2: 🔹 Molecules の作成

作業場所: 01_DesignSystem - Master ファイル

目的: Atomsを組み合わせた小さな機能単位を作成

📋 実装手順
  1. 複数のAtomsインスタンスを配置
  2. Auto Layoutで配置（Direction、Padding、Gapを設定）
  3. Instance Swapプロパティで内部Atomsを切り替え可能に
  4. Boolean Propertyでオプション要素の表示/非表示を制御
  5. 命名規則: {名称}/{タイプ}/{方向や配置}

💡 ポイント: Moleculesは意味のある最小機能単位です。検索フォーム、ナビゲーション項目、カード要素などが該当します。

Step 3: 🔷 Organisms の作成

作業場所: 01_DesignSystem - Master ファイル

目的: 独立した機能ブロックを構築

📋 実装手順
  1. MoleculesとAtomsのインスタンスを組み合わせ
  2. 複雑なAuto Layout構造を設計（ネストしたレイアウト）
  3. デバイス対応のバリアント: Desktop, Tablet, Mobile
  4. 状態管理: Expanded/Collapsed, Visible/Hidden など
  5. レスポンシブ対応: Min/Max Width を設定

💡 ポイント: Header、Footer、Sidebar、Navigation Barなど、複数のMoleculesで構成される複雑なコンポーネントです。

Step 4: 📋 Templates の作成

作業場所: 02_ProductUI ファイル（または 01_DesignSystem）

目的: レイアウト構造を定義

📋 実装手順
  1. 01_DesignSystemのOrganismsインスタンスを配置
  2. グリッドレイアウトやフレキシブルレイアウトを設計
  3. プレースホルダーコンテンツを使用
  4. コンテンツ領域のサイズ調整ロジックを実装
  5. Auto Layoutでレスポンシブ対応

💡 ポイント: Templatesはワイヤーフレームのようなもの。実際のコンテンツは入れず、構造だけを定義します。

Step 5: 📄 Pages の作成

作業場所: 02_ProductUI ファイル

目的: 実際のコンテンツを含む完成画面

📋 実装手順
  1. Templatesまたは直接Organismsインスタンスを配置
  2. 実際のテキスト、画像、データを入力
  3. コンポーネントのバリアントを実際の状態に設定
  4. 各種インタラクションの状態を表現（複数のPageで）
  5. デザインレビュー用にアノテーションを追加

💡 ポイント: ここで初めて実際のユーザーが見る画面が完成します。開発チームへの仕様書としても機能します。

Step 6: 📁 Files の管理

作業場所: Figmaプロジェクト全体

目的: 適切なファイル構成で管理

📋 ファイル構成
  1. 01_DesignSystem - Master
    • すべてのAtoms, Molecules, Organismsを管理
    • FlyonUIライブラリへの依存を集約
    • チームライブラリとしてPublish
  2. 02_ProductUI
    • 01_DesignSystemライブラリを有効化
    • Templates, Pagesを配置
    • FlyonUIライブラリは最終的に無効化

💡 ポイント: この2ファイル構成により、依存関係がシンプルになり、保守性が向上します。

🔗 依存関係と連携のまとめ

📊 依存の流れ
FlyonUI Library
    ↓ (Instance → Component)
01_DesignSystem - Master
    ├── Atoms (FlyonUIをラップ)
    ├── Molecules (Atomsを組み合わせ)
    └── Organisms (MoleculesとAtomsを組み合わせ)
        ↓ (Team Library)
02_ProductUI
    ├── Templates (Organismsを配置)
    └── Pages (実際のコンテンツ)
🎯 各レベルでのFlyonUI連携
  • Atoms: FlyonUIコンポーネントを直接ラップ、または自作
  • Molecules: ラップしたAtomsまたはFlyonUIコンポーネントを組み合わせ
  • Organisms: 自社のMoleculesとAtomsのみを使用（FlyonUIは間接的）
  • Templates/Pages: 自社のOrganismsのみを使用（FlyonUI依存なし）
⚡ バリアントとインスタンスの使い分け
概念 使用目的 例
Variant 同じコンポーネントの異なる状態 Button の Default/Hover/Active
Instance コンポーネントの実体を配置 Header内にLogo, Nav, User-Menuのインスタンス
Instance Swap インスタンスを動的に切り替え Stat Card内のIconを変更可能に

  • ✅ FlyonUIハイブリッド戦略との連携方法
  • ✅ バリアント、インスタンス、Instance Swapの使い分け
  • ✅ 下から上への構築順序とその理由
  • ✅ 依存関係の管理と保守性の向上方法
